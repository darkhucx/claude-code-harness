## 🔴 ultrawork の harness-review 強制メカニズム `cc:done`

### 背景 / 目的
- ultrawork がセッション継続（compact）後に harness-review をスキップして完了扱いにできてしまう問題が発生
- 原因: ルールはあるが強制力がない（物理的に破ることが可能）
- 参考: [Ralph](https://github.com/snarktank/ralph) の `<promise>COMPLETE</promise>` や `progress.txt` による状態永続化

### 問題の詳細
1. **セッション継続時のコンテキスト欠落**: compact 後に ultrawork フローの認識が失われる
2. **完了判定の誤り**: 「テスト全パス + ビルド成功 = 完了」と自己判断してしまう
3. **TodoWrite の完了マークにガードがない**: harness-review 実行有無をチェックする仕組みがない
4. **暗黙のルール**: 設計意図（review 必須）が強制力として実装されていない

### スコープ（やる）
- `ultrawork-active.json` に `review_status` フィールド追加（pending/passed/failed）
- Write/Edit 時に `review_status` を pending に戻す
- session-resume.sh + userprompt-inject-policy.sh で ultrawork モード時のフロー再注入
- ultrawork.md に明示的な強制ルールを追記

### 非スコープ（やらない）
- harness-review コマンド自体の変更
- /work コマンドへの影響
- Plans.md のマーカー運用ルール変更

### 受入条件（必ず測定可能に）
- [ ] `ultrawork-active.json` のスキーマに `review_status: "pending" | "passed" | "failed"` が含まれる
- [ ] ultrawork 完了処理が `review_status !== "passed"` で警告を出す
- [ ] session-resume.sh が ultrawork モード検出時に「harness-review 必須」を system-reminder で再注入
- [ ] ultrawork.md に「harness-review で APPROVE を得るまで完了処理不可」が明記
- [ ] harness-review 実行後に `ultrawork-active.json` の `review_status` が更新される

### 状態遷移図（Codex 承認済み）

```
開始 → pending
         ↓
    Write/Edit → pending (維持/リセット)
         ↓
    harness-review
         ↓
    ├── APPROVE → passed → 完了可能
    └── NOT APPROVE → failed → 修正 → Write/Edit → pending
```

### タスク（Claude Code 実装用）

#### Phase 1: 状態ファイルの拡張 `cc:done`
- [ ] `ultrawork-active.json` スキーマに `review_status` フィールド追加
- [ ] ultrawork.md のスキーマ定義を更新
- [ ] ultrawork 開始時に `review_status: "pending"` を設定する指示を追加

#### Phase 2: harness-review で review_status 更新 `cc:done`
- [ ] harness-review.md に APPROVE 時の `review_status=passed` 更新手順を追加
- [ ] harness-review.md に NG 時の `review_status=failed` 更新手順を追加

#### Phase 3: Write/Edit で review_status を pending に戻す `cc:done`
- [ ] track-changes.sh に ultrawork モード検出ロジック追加

#### Phase 4: セッション継続対応 `cc:done`
- [ ] session-resume.sh で `ultrawork-active.json` 存在時のフロー再注入
- [ ] userprompt-inject-policy.sh に一度だけの review 必須注入を追加（compact 対策）

#### Phase 5: TodoWrite 警告（補助） `cc:done`

## 🟢 Phase 39: Aivis Cloud API 統合（Remotion 動画ナレーション） `cc:完了`

### 背景 / 目的

- Remotion で生成した動画に音声ナレーションを追加したい
- Aivis Cloud API（https://api.aivis-project.com/v1）を使用してTTS生成
- シーン単位でナレーションを自動生成し、動画に統合

### スコープ（やる）

- `remotion/src/utils/aivis-client.ts` - Aivis API クライアント
- `remotion/src/utils/narration-generator.ts` - シーン説明からナレーション生成
- `remotion/src/hooks/useNarration.ts` - シーン音声管理フック
- `remotion/src/components/NarratedScene.tsx` - 音声付きシーンコンポーネント
- `remotion/src/HarnessPromoV6Narrated.tsx` - V6 ナレーション版

### 非スコープ（やらない）

- 既存シーンのビジュアル変更
- 音声合成モデルの選定（ユーザー指定）
- BGM/効果音の追加

### 受入条件（必ず測定可能に）

- [ ] `remotion/src/utils/aivis-client.ts` が存在し、Aivis API を呼び出せる

- [ ] 環境変数 `AIVIS_API_KEY` で認証できる

```
Base URL: https://api.aivis-project.com/v1

  "output_format": "wav",

}
```
