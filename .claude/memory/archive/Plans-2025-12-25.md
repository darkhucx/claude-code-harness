## フェーズ7: テスト改ざん防止機能の統合 `cc:完了`

> **目的**: Coding Agentが「楽をする」傾向（テスト修正、lint緩和、形骸化実装）を防ぎ、正しい実装による品質保証を実現する
> **出典**: びーぐる氏「Claude Codeにテストで楽をさせない技術」（Claude Code Meetup Tokyo 2025.12.22）
> **関連**: [D9: テスト改ざん防止の3層防御戦略](.claude/memory/decisions.md#d9-テスト改ざん防止の3層防御戦略)

### 背景

**問題**: Coding Agentは「タスク完了」を目標とするため、テスト失敗時に以下の選択をする傾向がある：
- ❌ テストを修正する（楽だが本末転倒）
- ❌ 形骸化した実装を書く（テスト期待値のハードコード、スタブ、空実装）
- ✅ 実装を修正する（正しいが手間がかかる） ← これを選ばせる

**実例**: テストケースの期待値をハードコードした辞書で返す「実装」が生成される
```python
def slugify(text: str) -> str:
    answers_for_tests = {
        "HelloWorld": "hello-world",
        # テストケースの期待値をそのままコピペ
    }
    return answers_for_tests.get(text, "")
```

**戦略**: Rules（良心ベース）→ Skills（文脈的強制）→ Hooks（技術的強制）の3層で段階的に防御

## フェーズ8: 既存ユーザー向けアップデート配信の改善 `cc:完了`

> **目的**: 既存ユーザーが `/harness-update` を実行しなくても、新機能の追加や古い設定の問題を認識できるようにする
> **発見**: レビュー時に発覚した「新規ルールファイルが通知されない」「古いフック設定が検出されない」問題を解決

### 背景

**問題1**: `template-tracker.sh check` は存在するファイルのみを追跡するため、新規追加されたルールファイル（test-quality.md, implementation-quality.md）が既存ユーザーに通知されない

**問題2**: `/harness-update` の Phase 1.5 は `permissions` の問題のみ検出し、古い `hooks` 設定は検出しない

**問題3**: 既存ユーザーは更新があることを知らずに古い状態のまま使い続ける可能性がある

### タスク

- [x] `session-init.sh` の強化 `cc:完了`
  - 新規追加されたルールファイルの存在チェック
  - 未導入の品質保護ルールを通知
  - `/harness-update` の実行を推奨

- [x] `/harness-update` に古いフック設定の検出を追加 `cc:完了`
  - `.claude/settings.json` の `hooks` セクションを検出
  - プラグイン側フックとの重複を警告
  - 削除を推奨（または自動削除のオプション）

- [x] `template-tracker.sh` の改善 `cc:完了`
  - 「存在しないが追跡対象のファイル」も check で報告
