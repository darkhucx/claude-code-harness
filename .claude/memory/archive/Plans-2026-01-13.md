> **目的**: harness-ui から Claude Code をプログラマティックに制御し、harness 思想に基づいた開発体験を UI 上で完結させる

## フェーズ21: Skills 公式仕様準拠（Progressive Disclosure パターン導入） `cc:完了`

> **目的**: 「スキルが見つからない」エラーを解消し、公式仕様に完全準拠
> **参照**: [Agent Skills Best Practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### 背景・問題点

**現在の構造（公式非準拠）**:
```
skills/impl/
├── SKILL.md                      # 親スキル（ルーター）
├── work-impl-feature/
│   └── doc.md                   # 子スキル（発見されない）
└── work-write-tests/
    └── doc.md
```

**公式仕様との差分**:
| 項目 | 公式仕様 | 現在の実装 | 問題度 |
|------|----------|-----------|--------|
| ネスト | 明示的に非推奨 | 2階層構造 | 高 |
| ファイル名 | 説明的な名前 | 汎用的な `doc.md` | 中 |
| 参照パス | `references/filename.md` | `subdir/doc.md` | 中 |

**エラーの原因**:
- Skill tool は SKILL.md のみ検出 → `doc.md` は発見されない
- 親スキル内の「doc.md を読め」指示が機能しない場合がある

---

### 改善策（Progressive Disclosure パターン準拠）

**After:**
```
skills/impl/
├── SKILL.md                              # 概要 + ナビゲーション
└── references/
    ├── implementing-features.md          # 詳細手順
    └── writing-tests.md
```

**SKILL.md の参照方法（公式パターン）**:
```markdown
## 機能詳細
**機能実装**: See [references/implementing-features.md](references/implementing-features.md)
**テスト作成**: See [references/writing-tests.md](references/writing-tests.md)
```

## フェーズ27: Codex モード統合（並列レビュー + コミット判定） `cc:TODO`

### 背景・要件

- harness-review は統合サマリーを表示するが、「コミットしていいか」の明確な判定がない

**解決策**:
- 全モードで「APPROVE / REQUEST CHANGES / REJECT」の明確な判定を出力
- Codex モード時は既存6観点を Codex に並列委譲（GPT 視点を最大活用）
- REQUEST CHANGES 時は Claude が修正 → 再判定のループ

    │
    └── Codex モード: Codex 委譲
            │
            ├── Claude オーケストレーション
            │       ↓
            │   ┌─────────────────────────────────┐
            │   │  Codex 並列起動（6+1観点）      │

            │   │  ├── Accessibility              │

## フェーズ26: ブラウザ自動化ツール整理（agent-browser 優先化） `cc:完了`

> **目的**: 乱立するブラウザ系ツール（Chrome DevTools MCP、Playwright MCP、dev-browser ルール）を整理し、agent-browser を優先使用する仕組みを構築
> **背景**: dev-browser ルールは存在するが実際には使われず、MCP ツールが無秩序に使われている状態

### 現状分析

**利用可能なブラウザ系ツール**:

| ツール | 種別 | 現在の利用状況 | 問題点 |
|--------|------|---------------|--------|
| `mcp__chrome-devtools__*` | MCP | 頻繁に使用 | 接続不安定、設定が複雑 |
| `mcp__plugin_playwright__*` | MCP | 時々使用 | セットアップが必要 |
| `SawyerHood/dev-browser` | Plugin | ほぼ未使用 | ドキュメントのみで実質死んでいる |
| `agent-browser` (CLI) | CLI | 未導入 | **新規導入済み** |

**問題の本質**:
- ルール（`.claude/rules/`）は「良心ベース」で強制力がない
- MCP ツールが先に見つかると、そちらが使われてしまう
- 明確な優先順位がシステムに組み込まれていない

---

### 解決策: 3層での agent-browser 優先化

```
┌─────────────────────────────────────────────────────────┐
│ 第1層: Rules（良心ベース）                               │
│   - ui-debugging-agent-browser.md.template              │
│   - 「agent-browser を優先使用せよ」と明記              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 第2層: Skills（文脈的強制）                              │
│   - troubleshoot/SKILL.md に agent-browser 優先を明記   │
│   - UI デバッグフローで agent-browser を最初に試す      │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 第3層: Hooks（システム強制）                             │
│   - PreToolUse で browser 系 MCP 使用時に警告          │
│   - 「agent-browser を先に試しましたか？」を表示        │
└─────────────────────────────────────────────────────────┘
```
